# ドキュメントレビュースキル

このスキルは、技術文書や仕様書などのドキュメントを体系的にレビューし、品質向上のための具体的なフィードバックを提供します。

## このスキルを使用するタイミング

- ドキュメントのレビューを依頼されたとき
- 仕様書や設計書の品質チェックが必要なとき
- 文書の網羅性や正確性を確認したいとき
- 参照情報との整合性を確認する必要があるとき
- システム間の影響を考慮したレビューが必要なとき

## レビュー観点

このスキルは以下の観点でドキュメントをレビューします:

### 1. 網羅性
- 必要な情報がすべて記載されているか
- 想定される質問に対する答えが含まれているか
- 手順やフローに抜け漏れがないか
- エッジケースや例外処理について言及されているか

### 2. 正確性
- 記載内容に事実誤認や矛盾がないか
- 技術用語が正しく使用されているか
- データ、数値、日付などが正確か
- 引用や参照が正確か

### 3. わかりやすさ
- 文章構造が明確で理解しやすいか
- 専門用語の説明が適切になされているか
- 図表やサンプルコードが効果的に使用されているか
- 対象読者に適した表現レベルか
- 論理的な順序で説明されているか

### 4. 参照情報との整合性
- 提供された参照ドキュメントとの矛盾がないか
- 関連する仕様や要件との整合性が取れているか
- 既存のドキュメントとの一貫性が保たれているか
- バージョン間の変更が適切に反映されているか

### 5. 機能面・システム面の観点
- 他システムへの影響が適切に考慮されているか
- 依存関係が明確に記載されているか
- パフォーマンスやセキュリティへの配慮があるか
- 運用・保守の観点が含まれているか
- スケーラビリティや拡張性について言及されているか

## レビュー手順

### ステップ1: 入力情報の確認

レビュー対象ファイルを受け取ります(必須)。

**参照情報の確認**:
- 参照情報(関連ドキュメント、仕様書、要件定義など)が提供されているか確認
- 提供されていない場合は、以下を質問:
  - 「このドキュメントのレビューに際して、参照すべき関連情報はありますか?」
  - 「関連する仕様書、要件定義、設計書などがあれば教えてください」
  - 「他システムとの連携や依存関係について確認すべき情報はありますか?」

### ステップ2: ドキュメントの読み込みと構造把握

1. レビュー対象ファイルを読み込む
2. ドキュメントの種類と目的を理解する(仕様書、設計書、手順書など)
3. ドキュメントの全体構造を把握する
4. 参照情報があれば、それも読み込む

### ステップ3: 観点別レビューの実施

各観点について順番にレビューを行い、以下の形式で整理:

```markdown
## レビュー結果

### 1. 網羅性
#### ✅ 良い点
- [具体的な良い点]

#### ⚠️ 改善提案
- [具体的な改善提案と理由]

### 2. 正確性
...

(以下、各観点について同様に記載)
```

### ステップ4: 優先度付けと総評

1. 発見した問題点に優先度を付ける:
   - 🔴 **重大**: すぐに修正が必要
   - 🟡 **中**: 改善が望ましい
   - 🟢 **軽微**: 余裕があれば改善

2. 総合評価と主要な推奨事項をまとめる

## 出力フォーマット

レビュー結果は以下の構成で出力します:


## ベストプラクティス

### 建設的なフィードバック
- 問題点を指摘するだけでなく、具体的な改善案を提示する
- 良い点も積極的に評価する
- 批判的ではなく、改善を支援する姿勢を保つ

### 具体性の重視
- 「わかりにくい」ではなく、「どの部分がどのようにわかりにくいか」を明確にする
- 抽象的な指摘は避け、具体的な行番号や該当箇所を示す
- 改善提案は実装可能なレベルまで具体化する

### コンテキストの考慮
- ドキュメントの目的と対象読者を常に意識する
- プロジェクトの制約や状況を考慮する
- 参照情報がある場合は、それとの関連性を重視する

### 優先順位の明確化
- すべての問題を同列に扱わない
- ビジネスインパクトや修正の緊急性を考慮する
- 限られたリソースで最大の効果を得られる改善を優先する

## AIアシスタント向け指示

このスキルが起動された際は、以下の手順に従ってください:

### 必須手順

1. **最初に必ず確認**:
   - レビュー対象ファイルのパスを確認
   - 参照情報の有無を確認
   - 参照情報が提供されていない場合は、必要性をユーザーにヒアリング

2. **ファイルの読み込み**:
   - Read ツールを使用してレビュー対象ファイルを読み込む
   - 参照情報があれば、それも読み込む

3. **体系的なレビュー**:
   - 5つの観点すべてについてレビューを実施
   - 各観点で「良い点」と「改善提案」の両方を記載
   - 必ず具体的な箇所(行番号など)を示す

4. **優先度付け**:
   - 発見した問題を重大度で分類
   - 🔴🟡🟢 のアイコンで視覚的に表現

5. **アクションの提案**:
   - 次にユーザーが取るべき具体的なアクションを提示

### 常に実施すること

- ポジティブなフィードバックと改善提案のバランスを取る
- 文化的・言語的な配慮を行う(日本語ドキュメントの慣習を尊重)
- レビュー結果は上記の出力フォーマットに従って整形する
- 具体的な行番号や該当箇所を示す
- 改善提案には「なぜ」その改善が必要かの理由も含める

### 決して実施しないこと

- 参照情報の確認をスキップしない
- 一部の観点のみでレビューを終了しない
- 抽象的な指摘だけで終わらない
- 批判的・否定的なトーンを使用しない
- ドキュメントの目的や対象読者を無視しない

## レビュー例

### 例1: API仕様書のレビュー

**入力**:
- レビュー対象: `api-specification.md`
- 参照情報: `requirements.md`, `system-architecture.md`

**レビューの焦点**:
- エンドポイントの網羅性
- リクエスト/レスポンスの正確性
- エラーハンドリングの記載
- 既存APIとの整合性
- 他サービスへの影響

### 例2: システム設計書のレビュー

**入力**:
- レビュー対象: `system-design.md`
- 参照情報: なし → ヒアリング実施

**ヒアリング内容**:
- 「この設計書に関連する要件定義書はありますか?」
- 「連携する他システムのドキュメントはありますか?」
- 「既存システムの設計書で参照すべきものはありますか?」

**レビューの焦点**:
- アーキテクチャの妥当性
- 非機能要件の考慮
- スケーラビリティ
- セキュリティ対策

### 例3: 運用手順書のレビュー

**入力**:
- レビュー対象: `operation-manual.md`
- 参照情報: `system-design.md`

**レビューの焦点**:
- 手順の完全性と順序
- エラー時の対応手順
- 前提条件と事後条件
- 実際の運用での実行可能性

## トラブルシューティング

### 問題: 参照情報が不十分
**対処**: ユーザーに追加の参照情報を依頼する。参照情報なしでレビュー可能な範囲を明確にする。

### 問題: ドキュメントの目的が不明確
**対処**: ドキュメントの目的、対象読者、使用場面をユーザーに確認する。

### 問題: レビュー観点が多すぎて時間がかかる
**対処**: ユーザーに特に重視する観点を確認し、優先順位をつけてレビューする。
