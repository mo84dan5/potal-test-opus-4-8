## Issue Workerの計画

### [o] セクション0 このIssueで何をやりたいか

**要件**: セーブ/ロード。保存は**文字列(セーブコード)をコピー**、**貼り付けてロード**。
将来は**画像で保存/画像からロード**したい。SOLID・「安全に忘れる」クリーンアーキ前提。

**設計の要(境界=文字列コード)**:
- **`GameSnapshot`(domain値)**: セーブ対象の直列化スナップショット。
  `version` / `worldId` / `player(x,y,z,yaw,pitch)` / `flags` / `completedEvents` / `props(id→x,z)`。
- **`SnapshotCodec`(domainポート, IF)**: `encode(snapshot)→string` / `decode(string)→snapshot`。
  実装はアダプタ `Base64JsonSnapshotCodec`(JSON+base64)。**domain は base64 等の実装を知らない(DIP)**。
- **`SaveService`(domainサービス)**: `capture(session)→GameSnapshot` / `restore(session, snapshot)`。
  集約 `GameSession` の状態を読み書き(プラットフォーム非依存・単一責務)。
- **UI/トランスポート(アダプタ)**: 今はクリップボード(コピー/貼り付けのテキストエリア)。
  将来の**画像**形式は、同じ「文字列(セーブコード)」を運ぶ別アダプタを足すだけ(SaveService/Codec は不変=安全に忘れる)。

「保存=文字列」を不変の契約にすることで、将来の画像保存は「文字列⇄画像」の新アダプタの追加で済む(OCP)。

### [o] セクション3 実装・テスト・デプロイ

#### 方針
- domain は文字列化の実装(base64)を持たない。Codec ポートをアダプタで実装(DIP)。
- セーブコードは将来の画像形式が運ぶ「中身」。契約を文字列に固定して拡張余地を残す(OCP)。
- ロード時は進行中イベント/会話を解除し、整合状態へ復元。

#### 結果
- domain/values/GameSnapshot.ts: `GameSnapshot`・`SnapshotCodec`(ポート)・`isValidSnapshot`・SNAPSHOT_VERSION。
- domain/services/SaveService.ts: capture(session)→snapshot / restore(session,snapshot)(イベント解除・プロップ位置含む)。
- adapters/persistence/Base64JsonSnapshotCodec.ts: ポート実装(`PW1.`+base64(encodeURIComponent(JSON)))。
- main.ts: SaveService/Codec 配線、セーブ/ロードパネル(コピー/貼付/ロード)、イベント中ボタン非表示・パネル中は操作ロック。
- index.html: 💾セーブボタン + パネル(textarea+ボタン群)。
- テスト: SaveService(capture/restore/round-trip/未知ワールド例外)、Codec(round-trip/空白/各種不正) → vitest 195件 全パス。ビルド成功。
- 仕様書: FN-016 新規、機能一覧/index 更新。
- デプロイ: main へ push → GitHub Pages Actions で自動デプロイ。
- 将来の画像形式: 「文字列(セーブコード)」を不変契約とし、画像に内包する別トランスポートを足すだけで対応可(SaveService/GameSnapshot/codec は不変)。
